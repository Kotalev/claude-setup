# Auto Tasks

An autonomous task queue for Claude Code. You drop a problem into a file, approve it, and Claude Code develops it in an isolated git worktree, reviews it, then verifies it in the browser against the Acceptance Criteria. `run` and `verify` are invoked manually (or wired to any external scheduler you like). Each task is a markdown file with YAML frontmatter; all state lives entirely in the filesystem.

- **Slash command:** `.claude/commands/auto-tasks.md` (`/auto-tasks ...`)
- **CLI behind it:** `scripts/auto-tasks/index.js` (Node, no external dependencies — built-in modules only)
- **Queue:** `tasks/inbox|processing|archive/` + `tasks/auto-tasks.config.json`

Principle: every mutation of a task goes through the CLI, never through manual edits. The directory a task sits in is the atomic lock (moved with `fs.rename`).

## Lifecycle

```
new ──(approve)──▶ for_dev ──(run)──▶ processing
processing ──(review ≥ 7)──▶ awaiting_verification
processing ──(error / review stuck)──▶ failed
awaiting_verification ──(verify)──▶ verifying ──(all ACs pass)──▶ verified (→ archive/)
verifying ──(any AC fails)──▶ not_verified
not_verified / failed ──(retry)──▶ for_dev
```

`failed` and `not_verified` stay in `processing/` (the worktree is kept for debugging). Only `verified` moves to `archive/`.

## Commands

Invoked as `/auto-tasks <subcommand> [args]`.

| Command | What it does |
|---|---|
| `install` | One-time setup: detects the stack and writes `tasks/auto-tasks.config.json`. **Run this first.** |
| `new` | Interactive stub (asks for title and test_url). |
| `new "<description>"` | Creates a task from a short description; greps for likely files and checks for related open tasks. |
| `new --from <path>` | Extracts and groups multiple tasks from a document (PDF review, list of problems). |
| `approve <id...>` | Moves task(s) from `new` to `for_dev`. |
| `list` | Table of counts by status. |
| `status <id>` | Details for a single task. |
| `run` | Claims all `for_dev` tasks, develops them in worktrees (subagents), runs the review loop. Invoked manually. |
| `verify` | Claims the oldest `awaiting_verification`, starts the apps from `config.verify.apps`, verifies via Chrome DevTools MCP, decides `verified`/`not_verified`. Invoked manually. |
| `retry <id>` | Returns a `failed`/`not_verified` task back to `for_dev` and deletes the old worktree+branch. |
| `cleanup` | Deletes tasks manually marked `status: commited`/`committed`, along with their worktree (the branch is preserved). |

### `new` options
- No argument → interactive stub.
- `"<description>"` → auto-extracts title + related files.
- `--from <path>` → batch ingest with grouping (union by shared file, module root, or `topic`).

`test_url` (relative) is optional — set it to the specific page that exercises the change for a direct open during verification. Leave it empty and the verify agent works out the right link itself from the task description.

## Configuration

`tasks/auto-tasks.config.json` is generated by `install` and **is committed** (shared setup). Read it with `node scripts/auto-tasks/index.js config`. Keys (with defaults):

| Key | Default | Meaning |
|---|---|---|
| `packageManager` | `npm` | Drives the `install`/`test` commands. |
| `install` | `<pm> install` | Install command run in the worktree. |
| `test` | `<pm> test` | Command the dev agent must get green. |
| `worktree.root` | `.claude/worktrees` | Where worktrees are created. |
| `worktree.branchPrefix` | `feature/` | Prefix for feature branches. |
| `moduleRoots` | `[]` | Globs for grouping by module (e.g. `apps/*/src/*`). |
| `projectRules` | `[]` | `.claude/rules/*.md` injected into the dev agent's prompt. |
| `reviewerAgent` | `elite-code-reviewer` | Subagent for the review loop (score ≥ 7 required). |
| `verify.enabled` | `apps.length > 0` | Whether verify is active. |
| `verify.mode` | `worktree` | `worktree` (starts apps from the worktree) or `checkout` (checks the branch out in the main repo — requires a clean tree). |
| `verify.envFile` | `.env` | Env file passed through the dotenv runner. |
| `verify.browserApp` | first app | Which app is the browser base URL. |
| `verify.apps` | `[]` | List of apps to start: `{name, cwd, start, port, portEnv?, portArg?, env?, health?, readyLog?}`. |

## Installing in a new project

### 1. Copy these files/directories from this repo

| From `project/` | Note |
|---|---|
| `scripts/auto-tasks/` | The whole CLI (`index.js`, `lib/`, `package.json`). `__tests__/` is optional. No external dependencies. |
| `.claude/commands/auto-tasks.md` | The slash command. |
| `tasks/` | The skeleton: `inbox/.gitkeep`, `processing/.gitkeep`, `archive/.gitkeep`, `README.md`. Do **not** copy other people's task files. |
| `.claude/rules/code-review.md` | Optional — defines the `elite-code-reviewer` expectation. If you don't have such an agent, change `reviewerAgent` in the config. |

Add to the new project's `.gitignore` (task files are local-only, the config is shared):

```gitignore
tasks/inbox/*
!tasks/inbox/.gitkeep
tasks/processing/*
!tasks/processing/.gitkeep
tasks/archive/*
!tasks/archive/.gitkeep
tasks/.run-log.jsonl
# tasks/auto-tasks.config.json is NOT ignored — it is committed.
```

### 2. Requirements
- **Node ≥ 20** (the CLI uses built-in modules only).
- Claude Code with **Chrome DevTools MCP** — only if you'll use `verify`.
- A subagent named `elite-code-reviewer` (or change `reviewerAgent` in the config).

### 3. Install command

From the root of the new project (must be a git repo), in Claude Code:

```
/auto-tasks install
```

This detects the stack (package manager, apps, ports, frameworks), asks about anything unclear, and writes `tasks/auto-tasks.config.json`. Only after that do `run`/`verify` work. Re-running is safe — it overwrites the config. Then you invoke `/auto-tasks run` and `/auto-tasks verify` manually whenever you want.
